# CLAUDE.md — рабочая память проекта

Этот файл читается в начале каждой сессии. Он фиксирует договорённости,
до которых мы дошли, чтобы они не пересобирались заново. Полные версии —
в `meta/`. Здесь — сжатая рабочая выжимка.

---

## Что это за книга

Учебник по веб-разработке для **абсолютных новичков**. Преемник по духу
к «книге 3» (формы 1С): тот же подход, тот же тип читателя.

Рабочее название — «Веб глазами древнего человека», поменять.

---

## Высший принцип (над всем остальным)

**Книга объясняет причинность, а не выдаёт рецепты.**

Не «делай так», а «это так, потому что иначе ломалось вот это; для того,
чтобы; из-за того, что». Инженерное мышление: ценность не в готовых
ответах, а в умении их вывести.

Рабочее правило проверки: если в тексте есть «надо / следует / правильно»
без «потому что» рядом — **это брак, переписать**. Читатель после
параграфа должен уметь не повторить рецепт, а объяснить своими словами,
почему так, а не иначе.

---

## Две линзы оптики

Каждый параграф смотрит на тему двумя глазами сразу.

**Линза 1 — Боль.** *Откуда* технология взялась. Каждая родилась как
ответ на невыносимость предыдущего шага. Это несущая ось: порядок
модулей = порядок болей. Цепочка из 10 болей — в `meta/spine.md`.

**Линза 2 — Три читателя.** *Для кого* технология. У страницы три
читателя, человек — последний в очереди:
1. **Поисковик — привратник.** Решает, увидят ли тебя вообще. Бинарен.
2. **Браузер — переводчик.** Решает, каким тебя увидят. Может переврать.
3. **Человек — заказчик.** Последний по очереди, но первопричина.
   Поисковик угождает не себе — своей модели человека. Накрутки
   обманывают всех и проигрывают.

Честная оговорка, проговаривать в книге: для сайтов за логином (админки,
вебклиент 1С) привратника нет — читатель сразу человек. Не оговорить —
подставить читателя-практика.

Реализация линзы 2: задаётся в §01-04, дальше живёт сквозной врезкой
«А кто это читает?» там, где ответ неочевиден (семантика, a11y, SSR,
метатеги, скорость).

## Метод подачи (тройная метафора + корни + боли)

Каждая технология вводится так:
1. **Древний предок** — как ту же задачу решали без компьютеров.
2. **Перенос в веб** — первая наивная попытка.
3. **Боли роста** — история переделок, каждая через конкретную проблему
   (примеры цепочек: авторизация, CORS, рендеринг, JS — в
   `meta/roots-and-pains.md`).
4. **Текущее состояние** — почему сейчас так и какие боли уже зреют.
5. **«Между строк»** — как это выглядит на сквозном примере.

Сквозная мысль книги: «YouTube в частности и соцсети в общем — это
наскальная живопись». Всё в вебе отвечает на древние потребности.

Три опорные метафоры, растут одна из другой по нужде:
- **Печатная машинка** — как сделать текст с оформлением (HTML, CSS).
- **Библиотека** — как донести его до Васи в Перми (HTTP, серверы, DNS).
- **Самопишущаяся книга** — почему всё усложнилось (БД, шаблоны, API,
  авторизация, JS).

## Сквозной пример: «Между строк»

Книжный интернет-магазин. Жанр (магазин с физическим товаром) выбран
потому, что только он тянет всю спину: каталог, корзина, заказы,
авторизация, оплата, доставка, интеграция с 1С.

**Стадии = спираль поверх оси болей.** Порядок глав определяет ось болей
и НЕ меняется под логику стройки сайта. Стадии — сквозная
демонстрационная линия. Стадия рождается, когда наступает её боль, а не
когда «по жизни так строят». Пример: заглушка «сайт в разработке» — это
первый HTML из Модуля 2, а НЕ глава про домены (домен прикручивается к
ней позже, в Модуле 11).

Стадии (= вехи = git-теги):

| Тег | Стадия | Рождается около |
|---|---|---|
| `stage/01-zaglushka` | «сайт в разработке», 1 HTML | M2 |
| `stage/02-lending` | одностраничный лендинг | M2–M3 |
| `stage/03-vitrina` | каталог из шаблонов + БД | M6–M7 |
| `stage/04-vitrina-blog` | + блог, SEO, три читателя | вплетается |
| `stage/05-magazin` | корзина, заказы, авторизация | M8 |
| `stage/06-zhivoy` | живой интерфейс (SPA) | M9 |
| `stage/07-integracii` | оплата, доставка, 1С | M10 |
| `stage/08-prod` | домен, хостинг, CI/CD | M11 |

Читатель делает `git checkout stage/03-vitrina` и видит состояние кода
ровно на том витке, до которого дочитал. Код примера — в `example/`.

## Стиль: Перельман пишет учебник по Каймину твоим голосом

- **От Перельмана:** заход через парадокс/эпизод, технология как
  разгадка, живые истории, право удивляться, автор в тексте.
- **От Каймина:** жёсткая структура, определения, контрольные вопросы,
  звёздочные задачи, резюме как повторение.
- **Твой голос:** сарказм без злобы, прагматизм педагога-практика,
  самоирония, короткие реплики, бытовые метафоры.
- **Не наше:** Киселёвская сухость, чистый научпоп без задач, блогерский
  тон с «лайк-подписка».

Полная стилевая записка — `meta/style-guide.md`.

## Структура параграфа (жёсткая)

1. Заход (древний предок / живая ситуация)
2. Боль
3. Решение (без «потому что» нельзя)
4. Разбор + «Между строк»
5. Резюме (5–8 строк)
6. Закрепление: 3 контрольных вопроса (на «почему», не на память)
   + 1–2 упражнения

Объём: 8–12 тыс. знаков основного текста. 30–45 мин чтения + 15–30 мин
упражнений = одна посиделка. Больше — это два параграфа.

Новых терминов в параграфе — не больше 3, каждый с переводом на
человеческий. Врезки: «Соври мне», «Историческая справка», «А вот в 1С»,
«А кто это читает?», «Внимание!», «Если копнуть».

## Конвенции репозитория

- **Моно-репо:** книга + код примера вместе.
- **Файл на параграф.** Имена: транслит, без точек, `NN-MM-slug.md`
  (напр. `02-03-html-skobki.md`).
- **`book.yml`** — единственный источник порядка глав для сборки.
  НЕ редактировать список `files` руками: правится манифест в
  `scripts/scaffold.py`, затем скрипт перегенерирует.
- **`scripts/scaffold.py`** не перезаписывает существующие .md
  (написанный текст не затирается). `book.yml` пересобирается всегда.
- **`meta/`** в репозитории (кухня видна намеренно). `meta/archive/` —
  как мы пришли (исходный план, майндмэп).
- **Ночные сборки:** `.github/workflows/nightly.yml` → EPUB/PDF/HTML
  через pandoc по `book.yml`. Артефакты в `dist/` (gitignored).
- **Статусы параграфа** во front-matter: `todo → draft → review → done`.

## Открытые вопросы

- **Лицензия.** Монорепо = двойная: текст (предлагается CC BY-NC-SA)
  + код примера (предлагается MIT). Финально не выбрано — см. `LICENSE`.
- **Финальное название книги.** Сейчас рабочее.
- **Объём.** 75 параграфов — верхняя граница. Цель к финалу — ужать до
  60–65 слиянием и выносом в звёздочные врезки.

## Чего НЕ делать

- Не вводить технологию фразой «принято делать так».
- Не пересказывать документацию и не цитировать спецификации/RFC.
- Не писать «как известно», «очевидно», «легко видеть».
- Не оставлять параграф без упражнения.
- Не лезть в нюансы раньше времени (в первом упоминании HTTP — никаких
  HTTP/2, multiplexing).
- Не менять порядок модулей под логику стройки сайта (ось — боль).
- Не редактировать список `files` в `book.yml` руками.
